# 분석 엔진을 Python에서 Kotlin(Spring Boot)으로 이관한 사례

## 배경

CAPD의 분석 엔진(추세 분석·이상치 탐지·상관관계 분석·EDA, `run_all_tasks`로 묶여 있는 4개 태스크)은 원래 순수 Python 함수로 작성되어 있었고, AI 서버(`ai/tools/analytics.py`)와 백엔드(`backend/app/services/analytics_engine.py`) 두 곳에 같은 로직이 복사되어 있었다. 이 프로젝트는 여기서 한 걸음 더 나아가, 같은 계산 로직을 완전히 다른 언어 스택(Kotlin + Spring Boot)으로 옮겨도 결과가 한 치의 오차 없이 동일하다는 것을 실제로 증명하고, 그 서비스를 인증·캐시까지 갖춘 실서비스급 완성본으로 만들어보는 것을 목표로 삼았다.

단순히 "포팅했다"는 결과보다, 포팅 과정에서 마주친 언어 간 수치 재현성 문제와 그것을 어떻게 검증 가능한 형태로 확인했는지가 이 작업의 핵심이다.

## 접근 — 두 단계로 나눠 진행

### 1차: 순수 계산 로직 포팅 + 정합성 검증

먼저 인증이나 캐시 없이, 계산 로직 자체가 Python과 완전히 같은 결과를 내는지부터 확인하는 슬라이스를 만들었다.

- `DataEngineering.kt` — 원본 일별 데이터를 회차별 교환 기록에서 25개 컬럼짜리 Daily Model Row로 변환하는 로직(`build_daily_model_row`)을 그대로 포팅.
- `AnalyticsTasks.kt` — 추세 분석, z-score/robust z-score 기반 이상치 탐지, Spearman 상관관계, EDA 4개 태스크(`run_all_tasks`)를 포팅.
- `GET /api/v1/analytics/patients/{patientId}?window=7~90` — 백엔드의 동일 엔드포인트를 JdbcTemplate 기반 직접 쿼리로 재현.
- 정합성 테스트(JUnit5, `ParityTest.kt`) — Python이 미리 계산해 둔 결과와 Kotlin 출력을 비교.

이 단계에서는 "로직이 완전히 같다"는 것 하나만 증명하는 데 집중했고, 실제 서비스로 노출할 수 있는 상태는 아니었다.

### 2차: 인증 + 접근권한 + 캐시 이관, 실서비스급 완성본으로

1차에서 의도적으로 제외했던 부분을 마저 옮겨서 실제로 배포 가능한 완성본으로 만들었다.

- **JWT 인증** — 백엔드(python-jose)가 발급한 HS256 액세스 토큰을 검증하는 컴포넌트. 표준 JWT 라이브러리 대신 HMAC-SHA256을 직접 계산해 서명을 비교하도록 구현했다(아래 "기술적으로 까다로웠던 것들" 참고).
- **담당의-환자 접근권한** — 의사 role이 아니면 403, 담당 이력이 없으면 403, 과거 담당이면 담당 종료일 이전 기록만 조회하는 로직을 백엔드와 동일하게 재현.
- **Silver/Gold 캐시** — 온디맨드로 계산한 결과를 캐시 테이블에 읽고 쓰는 로직을 백엔드와 1:1 동일한 쿼리·충돌 처리 규칙으로 포팅.

이 단계까지 마치니 인증 없이 호출하면 401, 권한이 없으면 403이 뜨는 정상적인 API 서비스가 되었다.

## 기술적으로 까다로웠던 것들

### 언어 간 수치 재현성 — Python의 `round()`를 자바/코틀린에서 재현하기

Python의 `round(x, n)`은 correctly-rounded round-half-to-even 방식이라, 자바/코틀린의 기본 반올림과 결과가 다를 수 있다. 이 차이를 없애기 위해 `BigDecimal(double).setScale(n, RoundingMode.HALF_EVEN)`로 반올림 유틸리티를 직접 구현했다. 여기서 중요한 포인트는 `BigDecimal.valueOf(double)`이 아니라 `BigDecimal(double)` 생성자를 써야 한다는 것이다 — `valueOf`는 내부적으로 `Double.toString()`을 거치기 때문에 이진 부동소수점의 정확한 값과 달라지고, 그 결과 미묘하게 다른 반올림 값이 나올 수 있다. 이 방식으로 바꾼 뒤 fixture 비교가 오차 없이 정확히 일치했다.

자연어 서술 문장(`statement` 필드) 안에 들어가는 숫자는 Python `str(float)`(David Gay's dtoa 알고리즘)과 코틀린 `Double.toString()`(당시 JDK 버전 기준)의 포맷팅 알고리즘이 극단적인 케이스에서 다를 수 있어, 정합성 테스트에서는 이 필드를 의도적으로 비교 대상에서 제외했다. 실제 판정에 쓰이는 수치·분류 필드는 전부 엄격 비교 대상이므로 로직 정합성 자체는 충분히 보장된다.

### JWT 검증 라이브러리의 숨은 제약

표준 JWT 라이브러리(JJWT 등)는 HS256 알고리즘에 256비트(32바이트) 미만의 키를 거부한다. 그런데 백엔드가 사용하는 Python 라이브러리(python-jose)는 그런 최소 길이를 강제하지 않는다. 즉 백엔드의 실제 시크릿 키 길이에 따라, 표준 라이브러리로 검증을 구현하면 Python에서는 정상 발급된 토큰이 Kotlin 쪽에서는 라이브러리 자체 예외로 거부당하는 상황이 생길 수 있었다.

이를 해결하기 위해 표준 라이브러리를 쓰지 않고 HMAC-SHA256을 직접 계산해서 서명을 비교하는 방식을 택했다. 새 패키지 의존성 없이 표준 크립토 API만으로 구현했고, 검증 규칙(서명 유효성·만료 시각·토큰 타입·subject 존재 여부)은 백엔드의 토큰 디코딩 로직과 동일하게 맞췄다.

### 크로스랭귀지 정합성 테스트 전략

같은 언어(Python↔Python) 사이의 정합성 테스트라면 시드를 고정한 난수 생성기를 공유해서 "같은 입력 → 같은 기대값"을 코드로 표현할 수 있다. 하지만 Kotlin은 Python의 PRNG 알고리즘을 재현할 수 없어서 이 방식이 통하지 않았다.

그래서 fixture 방식으로 전환했다. Python 쪽에서 시드 고정 합성 데이터를 생성하고, Python 원본 로직으로 계산한 실제 출력을 JSON 파일로 고정해 리포지토리에 커밋해 둔다. Kotlin 테스트는 같은 입력을 Kotlin 포팅본에 먹여서 이 JSON과 비교한다. 이렇게 하면 두 언어의 난수 생성 방식이 달라도, "같은 입력에 대해 같은 출력이 나오는가"라는 질문에 정확히 답할 수 있다.

## 로컬 빌드 환경 구축

이 작업은 별도의 로컬 개발 환경 없이 격리된 실행 환경 안에서 진행되었는데, 여기에는 몇 가지 제약이 있었다.

- 관리자 권한 없이 JDK 17과 Gradle을 설치해야 해서, 패키지 매니저 대신 공식 배포 바이너리를 직접 내려받아 홈 디렉터리에 압축을 풀어 사용했다.
- Gradle wrapper를 생성하는 표준 태스크가 실행 시간 제한 때문에 매번 타임아웃되어서, Gradle 배포판에 이미 포함된 wrapper jar를 직접 추출하고 wrapper 스크립트는 공개된 특정 버전 태그에서 그대로 받아오는 방식으로 우회했다.
- 소스가 위치한 마운트 폴더가 파일 삭제·재작성을 허용하지 않아 증분 빌드가 실패하는 문제가 있어서, 소스는 마운트된 위치에 두고 실제 빌드·테스트는 별도 스크래치 디렉터리에 동기화해서 실행했다.

이 방식으로 정합성 테스트 실행, JAR 패키징, 실제 JAR 기동(스프링 컨텍스트가 새로 추가한 컴포넌트까지 정상적으로 로드되는지)까지는 로컬에서 확인했다. 다만 이 환경에는 데이터베이스가 없어서, 실제 캐시 조회·쓰기 쿼리나 접근권한 조회 쿼리가 운영 DB에 대해 정상 동작하는지는 별도로(실제 데이터베이스가 연결된 환경에서) 확인이 필요했다.

## 검증 범위와 한계

- 순수 계산 로직(`ParityTest.kt`, 4개 테스트) — fixture와 완전히 일치, 자동화된 테스트로 지속적으로 확인 가능.
- 인증·접근권한·캐시 로직 — 로컬 환경에 DB가 없어 통합 테스트를 아직 추가하지 못했다. 이 부분은 다음 이터레이션 과제로 남아 있다.

## 배운 점

언어를 옮기는 작업에서 가장 까다로운 부분은 "로직을 다시 쓰는 것"이 아니라 "숫자가 정말 똑같이 나오는지 증명하는 것"이었다. 반올림 알고리즘, 부동소수점 문자열 변환, 암호 라이브러리의 세부 제약처럼 평소에는 신경 쓰지 않는 지점들이 실제 크로스랭귀지 이관에서는 전부 잠재적인 불일치 원인이 된다. 이번 작업에서 얻은 실용적인 결론은, 같은 언어 간에는 입력 생성기를 공유하는 것으로 충분하지만 다른 언어 간에는 골든 fixture를 고정해서 비교하는 방식이 훨씬 신뢰할 수 있다는 것이다.
